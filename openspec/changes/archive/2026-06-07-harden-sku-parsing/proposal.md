## Why

当前 `parse_dp_code` 在去掉 `gx-` 前缀后会**无条件砍掉末尾 3 个字符**。这对"带 3 位后缀的 SKU"（如 `gx-b0dfxqwpps149`）是对的，但对两类合法输入会误伤：

- **裸 ASIN**（无 3 位后缀，如 `b0dfxqwpps`）会被砍成 7 位 → 误判为格式错误。
- **结尾恰为数字的 10 位 ASIN**（如图书 ISBN 型 `4873115655`）会被砍掉真实字符 → 误判为格式错误。

随着 SKU 来源变多（人工粘贴、历史导出、第三方表格），上述输入会真实出现，需要把解析规则加固为**长度感知**的健壮规则，既忠于"去前缀 / 去 3 位数字后缀 / 否则不变"的本意，又不误伤合法 ASIN。

## What Changes

- 用**长度感知规则**替换"无条件砍末尾 3 位"：
  - 去 `gx-` 前缀（大小写不敏感），与现状一致。
  - 去前缀后若为 **13 位且末 3 位是数字** → 砍掉末 3 位（ASIN + 3 位后缀）→ 得 10 位。
  - 去前缀后若为 **恰好 10 位** → 原样保留（裸 ASIN，即使结尾是数字）。
  - 其余长度 → 标记 `FormatError`（与现状一致的错误状态）。
  - 末尾大写、`^[A-Z0-9]{10}$` 校验、去重逻辑保持不变。
- 补充覆盖新规则的单元测试（裸 ASIN、结尾数字的 ASIN、大写前缀、非 13/10 长度）。
- 非破坏性变更：`RowResult` 数据模型、命令/接口签名、前端均不变。

## Capabilities

### New Capabilities
- `sku-parsing`: 将一行原始 SKU 文本解析为 `(sku, dpCode, ASIN)`，含前缀剥离、长度感知的后缀剥离、ASIN 校验、按行去重与非法标记。

## Impact

- 代码：`src-tauri/src/sku.rs`（`parse_dp_code` 及其单测）；逻辑同时被桌面端与后续 Web 端复用。
