## Context

`src-tauri/src/sku.rs::parse_dp_code` 现状：trim → 去 `gx-` 前缀（大小写不敏感）→ **无条件 `body[..len-3]`（砍末尾 3 位）** → 大写 → 校验 `^[A-Z0-9]{10}$`。

约束与事实：
- Amazon ASIN 恒为 **10 位** 字母数字。
- 内部 SKU 的典型形态为 `gx-<asin><3 位数字后缀>`，去 `gx-` 后为 **13 位**。
- 真实输入会出现"裸 ASIN"（10 位，无后缀）与"结尾为数字的 ASIN"（如图书 ISBN-10 全数字）。
- 该解析逻辑为纯 Rust、无框架依赖，桌面端与后续 Web 端共用，必须稳。

## Goals / Non-Goals

**Goals:**
- 用一条**确定性、长度感知**的规则覆盖：带前缀/不带前缀 × 带后缀/不带后缀 的全部组合。
- 不误伤裸 ASIN 与结尾为数字的 ASIN。
- 保持数据模型、命令签名、去重与错误状态语义不变。

**Non-Goals:**
- 不引入对 Amazon 的联网校验（不验证 ASIN 是否真实存在）。
- 不支持一行多个 SKU、不支持 CSV/列解析（仍是按行一个 SKU）。
- 不改变 `gx-` 之外的其他潜在前缀（本次只认 `gx-`）。

## Decisions

### 决策 1：用"长度感知"替代"纯后缀匹配"

候选方案：

| 方案 | 规则 | 裸 ASIN `b0dfxqwpps` | 结尾数字 ASIN `4873115655` | 复杂度 |
|---|---|---|---|---|
| A. 现状 | 去前缀后无条件砍末 3 位 | ❌ 砍成 7 位 | ❌ 砍成 7 位 | 低 |
| B. 字面规则 | 结尾是 3 位数字就砍 | ✅ 不砍 | ❌ 误砍 | 低 |
| **C. 长度感知（选中）** | 13 位且末 3 位数字→砍；10 位→保留；其余→交校验 | ✅ 不砍 | ✅ 不砍 | 低 |

选 **C**：因为 ASIN 长度恒为 10、带后缀恒为 13，长度天然消歧；C 是唯一对全部已知合法输入都正确的方案，且实现复杂度与 A/B 同级。

派生伪代码：
```
s = trim(line)
if s.len() >= 3 && s[..3].eq_ignore_ascii_case("gx-") { s = s[3..] }
if s.len() == 13 && s[10..].chars().all(|c| c.is_ascii_digit()) {
    s = s[..10]
}
// s.len() == 10 的裸 ASIN 不做任何剥离；其余长度原样进入校验
asin = s.to_uppercase()
valid = asin.len() == 10 && asin.chars().all(|c| c.is_ascii_alphanumeric())
```

### 决策 2：`dpCode` 取剥离后、未大写的主体

保持与现状一致：`dpCode = 剥离后的小写主体`，`asin = dpCode.to_uppercase()`，CSV 与 UI 展示不变。

### 决策 3：非 10/13 长度一律判 `FormatError`

去前缀后落在 11–12 位或 >13 位（且不满足砍 3 位条件）的输入，不做猜测性剥离，直接判格式错误。宁可让用户看到明确错误，也不静默产出错误 ASIN。

## Risks / Trade-offs

- [极端歧义：13 位输入恰好"前 10 位是合法 ASIN 且末 3 位是数字"，但用户本意是别的 13 位编码] → 业务上 13 位带 3 位数字后缀就是既定 SKU 形态，按既定语义砍 3 位是正确解释；无法也无需进一步区分。
- [图书 ISBN-13 等 13 位纯数字被误当作"ASIN+后缀"] → 当前业务只处理 Amazon dp/ASIN，不涉 ISBN-13 输入；如未来需要，再加来源标识。属已知边界，本次不处理。
- [规则变更改变既有行为] → 对 `ids.txt` 全部 6 条与现有用例结果完全一致（带后缀场景不变），仅扩大了可接受输入集合，无回归风险；新增单测固化边界。
