## Context

抓取内核为纯 Rust：`region.rs`（HTTP 会话、地区设置、页面解析）、`scraper.rs`（并发引擎 + 重试 + 限速）、`config.rs`（常量与 URL 构造）、`models.rs`（数据形状）。当前取价链路为「设日本邮编 `150-0001` → `GET /dp/{ASIN}?th=1` → 取第一个含 `￥/¥` 的 `a-offscreen`，兜底用 `a-price-whole`+`a-price-fraction` 拼接」。该链路在本机/印尼网络下验证通过，但迁移到香港服务器后出现取价错误。

**实测证据**（抓取器出口 IP 在美国，可复现海外 IP 的货币本地化）：

| 来源 | 返回 | 说明 |
|---|---|---|
| `/s?k=b08ckgrhlf` | 单一价 `USD 36.20`（≈¥5,287） | 搜索命中真实 buybox 特价；但币种被本地化 |
| `/dp/B08CKGRHLF?th=1&psc=1` | `価格: ￥5,665　￥11,151` | 商品页含多个数、且与真实 buybox（¥5,287）不符 |
| 搜索结果尾部 | 一堆 `Sponsored` ELP 摄像头 | 必须按 `data-asin` 精确匹配以过滤广告 |

约束：单用户；服务器在香港（数据中心 IP）；优先不引入代理，先用 cookie 锁 + 自检暴露验证可行性。

## Goals / Non-Goals

**Goals:**
- 抓到的价格是**日元**且为**真实 buybox 特价**（与 `?th=1&psc=1` 一致）。
- 取价错误时**宁可显式失败**，不静默返回错误数字。
- 大批量抓取具备宏观限速以降低反爬风险。
- 远程部署后能**即时判断**日元锁是否生效（无需先抓一批才发现全错）。

**Non-Goals:**
- 不在本 change 引入代理/VPN（保留入口，作为后续 fallback change）。
- 不做多币种支持/汇率换算（目标就是日元）。
- 不做抓取结果持久化、不改认证/部署。
- 批量大小/冷却时长本次**写死**（100 / 30s），暂不做成可配置。

## Decisions

### 决策 1：货币锁定 —— 持久化 cookie jar + 防覆盖 + 请求级断言
- 将 `Client::builder().cookie_store(true)` 改为 `.cookie_provider(Arc<reqwest::cookie::Jar>)`，持有 jar 句柄，预置 `i18n-prefs=JPY`、`lc-acbjp=ja_JP`（域 `.amazon.co.jp`）。
- `init` 完成后**重新写回** JPY 偏好，抵御首页响应 `Set-Cookie` 把偏好改成按地理位置的币种。
- 取价请求（搜索/商品页）再显式断言一次，双保险。
- **为什么**：现状把 `i18n-prefs=JPY` 作为单次请求的手写 `Cookie` header，既不进 jar、又不防覆盖，所以「设了却没用」。
- **备选**：仅靠日本配送地址（不够，地址影响配送不直接决定显示币种）；URL 参数强制币种（无可靠参数）。

### 决策 2：取价主路径 —— 搜索页优先 + 商品页兜底
- 主：`GET /s?k={asin}` → 遍历 `div[data-component-type="s-search-result"]`，取 `data-asin` 等于目标 ASIN（忽略大小写）的卡片，读其 `.a-price .a-offscreen` 原文。
- 兜底：搜索 0 命中 → `GET /dp/{asin}?th=1&psc=1`（新增 `psc=1` 以锁定具体变体、消除区间）。
- **为什么**：实测搜索页命中真实 buybox 特价且为单一价；精确 `data-asin` 匹配天然过滤 Sponsored 广告；商品页兜底覆盖未被搜索索引的 ASIN。
- **备选**：只用搜索（个别 ASIN 无搜索结果会丢价）；只给商品页加 `psc=1`（实测仍可能取到非 buybox 的数）。

### 决策 3：币种校验 —— 读真实符号，非日元快速失败
- 解析返回从 `(text, value)` 扩展为 `(text, value, currency)`；`currency` 由页面**真实**符号判定（如 `a-price-symbol` 文本或 `a-offscreen` 原文中的货币标记），**不得**用系统自己 `format!("￥…")` 拼出来的字符串判定。
- 仅 `￥`（全角，JPY）记为成功；`$`/`HK$`/`€`/半角 `¥` 等记真实币种并标失败。
- 非日元属 IP 地理问题，**重试同一请求无意义** → 跳过重试、直接失败，错误信息写明检测到的币种。
- `RowResult.currency` 真正赋值（此前恒为 `"JPY"`）。
- **备选**：静默换算（不符合「要日元真值」目标且无可靠实时汇率）。

### 决策 4：大批量分批限速
- 在 `scrape_rows` 之上加宏观分批：实际抓取条数 > 100 时按 100 切块，块间 `sleep 30s`。
- 冷却用 `tokio::select!` 同时监听 `cancel_flag`，取消能立即打断 sleep。
- 进度按**总量**汇报；进入冷却时通过进度事件的阶段字段通知前端（见决策 5）。
- 常量写死：`BATCH_SIZE = 100`、`BATCH_COOLDOWN = 30s`。
- **为什么**：数据中心 IP 反爬更严，宏观停顿比单纯提高 per-request 间隔更能模拟人类节奏。

### 决策 5：进度可观测性 —— 进度事件增加阶段
- `ScrapeProgress` 增加可选阶段信息（如 `phase: scraping | cooling`、可带剩余冷却秒数/批次序号），前端据此显示「批次 x/y 完成，冷却 30s…」，避免冷却期看起来像卡死。

### 决策 6：自检币种感知
- `self_check` 返回检测到的币种与样例价；会话状态条显示如「自检通过：B0DFXQWPPS = ￥1,980 (JPY)」或「检测到 USD，货币未锁定」。
- **为什么**：这是「cookie 优先、不行再上代理」路线的反馈灯，部署即见效果。

### 决策 7：复制/导出保留小数
- 前端复制由 `replace(/[^\d]/g,"")` 改为保留小数点；当 `priceText` 含小数时优先用清洗后的文本，否则用 `priceValue`。
- **为什么**：防御性；锁定日元后通常为整数，但避免「有小数被吃掉」回归。

## Risks / Trade-offs

- [海外 IP 下 cookie 不一定 100% 压住货币] → 先上 cookie 锁 + 币种校验 + 自检暴露；若自检仍非日元，再以独立 change 引入日本节点代理（已保留 `reqwest` 代理入口）。
- [搜索页反爬比商品页更敏感] → 配合分批限速 + 既有 governor 限速（≤3/s）+ 随机抖动；搜索失败回退商品页。
- [Amazon 搜索/商品页 DOM 结构变动] → 选择器集中在 `region.rs`，且搜索按 `data-asin` 锚定卡片、商品页保留多级兜底；解析对单个选择器失效有降级。
- [搜索按关键字可能命中相关品而非目标] → 强制 `data-asin` 精确匹配，不匹配即视为 0 命中并兜底，杜绝取错品价格。
- [分批冷却拉长总时长] → 100 条 ≈ 33s（3/s）+ 30s 冷却 ≈ 每批 ~63s；1000 条约 10 分钟，单用户可接受。

## Migration Plan

1. `region.rs`：切到 `cookie_provider(Jar)`，预置/重置 JPY；新增搜索抓取与解析；`extract_price` 返回币种；商品页兜底带 `psc=1`。
2. `models.rs`/`config.rs`：`currency` 赋值、`ScrapeProgress` 阶段字段、批量常量与搜索 URL。
3. `scraper.rs`：分批 + 可取消冷却；非日元快速失败；`self_check` 币种感知。
4. 前端：进度阶段 UI、复制保留小数、币种展示。
5. 部署到香港服务器，看自检状态条判断日元锁是否生效；用样例 ASIN 端到端验收（目标 `B08CKGRHLF` = ¥5,287）。
- **回滚**：解析与会话改动集中在 `region.rs`/`scraper.rs`，可单独回退到当前商品页链路。

## Open Questions

- `psc=1` 对「SKU 解出的 10 位 ASIN 实为父 ASIN」的情形是否仍可能取到区间？（取决于用户数据中父/子货号比例；当前以搜索页为主已大幅缓解，商品页仅兜底。）
- 币种符号映射的覆盖面（半角 `¥` 与人民币的歧义）——实现期以「仅全角 `￥` 判为 JPY、其余一律视为非日元失败」从严处理。
